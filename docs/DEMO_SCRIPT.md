## 0. 制作方針

### 0.1 動画の目的（2種類）

| 動画 | 目的 | 長さ | 視聴者 |
|---|---|---|---|
| **A. 提出用デモ動画** | Proto Pedia / Google Form 提出。一次・二次審査で見られる | 3 分以内 | 審査員（運営事務局・外部有識者） |
| **B. 最終ピッチ** | 最終ピッチ会場 (8/19) で生披露 | 5-7 分 | 審査員 + 観客 (Google渋谷) |

両者で同じ素材を活用し、A は短縮版、B はライブ操作含む拡張版にする。

### 0.2 編集原則

1. **冒頭3秒で興味を掴む**（痛みのある問い）
2. **15秒で「これは何か」を理解させる**（一言コンセプト）
3. **中盤は体験デモ**（実 UI の操作のみ。モック・想像図は使わない）
4. **技術は「自律性が見える画面」で語る**（アーキ図の静止画は最小限）
5. **主張はすべて実装済みの事実**（未実装・未使用技術への言及は禁止。審査基準⑤「主張と実体の整合」対策）
6. 字幕を必ず焼き込む（審査員の多くが音声OFFで見る想定）

### 0.3 ツール

- 画面録画: OBS Studio / ブラウザ録画
- 編集: DaVinci Resolve (無料) or CapCut
- ナレーション: 自分の声 (Yuji)、BGM: YouTube オーディオライブラリ
- 実演 URL: `https://citify-web--citify-dev.asia-east1.hosted.app/`

### 0.4 事前セットアップ（撮影前日までに必ず）

- [ ] **データ鮮度**: Cloud Scheduler 5 job を resume（`./toggle-schedulers.sh resume` 相当）し、必要なら `publish-all` で議題を追加投入
- [ ] **フィード用ペルソナ**: `/municipalities` で **岡山市(33100)・札幌市(01100)・松山市(38201)** を登録（データ最多の 3 都市。「あなたの街」タブが確実に埋まる）
- [ ] **Watcher 用の事前 seed（レイテンシ対策・必須）**: プリセット3種は移住候補がデータ最多3都市に統一済み。**撮影当日に**ブラウザで **3 プリセットを1回ずつ選んで `/agent` を開く**→ キャッシュ無しなら自動で調査が起動する（③）ので、そのまま **各 1〜2 分待って結論が出た状態にしておく**。これで本番録画時は `/agent` を開くと"即結論"が表示される（実処理の待ちを録画に映さない）。3 persona 分を順に温める。

## 2. 最終ピッチ (5-7 分版) — 8/19 用

3 分デモに以下を追加:

| 追加 | 内容 |
|---|---|
| 前 30 秒 | 自己紹介 + 当事者性（上京 5 年、区議会を一度も読んだことがなかった） |
| 中 60 秒 | ライブ操作（Watcher の分析実行を目の前で。キャッシュ済ペルソナをバックアップに） |
| 後 90 秒 | 技術深掘り 1 本: **Watcher の自律性設計**（下記） |
| 終 45 秒 | 展望 + Q&A 準備（Watcher 非同期実行の遅延に備え 15 秒バッファ） |

### 技術深掘りスクリプト（Watcher の自律性）

> ここだけ 90 秒、技術の話をさせてください。
>
> Watcher は最初、完全自律型 — エージェントが毎ステップ次の行動を LLM で決める設計にしました。
> 動きはしたが、本番では逐次実行でレイテンシが破綻し、モデルの癖でツール選択も偏った。
>
> そこで「**計画は LLM、実行は並列**」に再設計しました。プランナーが最初に調査計画を立て、
> ツール呼び出しは並列で走らせ、結論の前に自己検証を挟む。
> カバレッジの床を保証するガードレールも入れています。
> 自律性は「全部 LLM に任せること」ではなく、**判断と実行の役割分担を設計すること** — これが 7 週間で得た結論です。

**［ここで画面を2分割：左に Watcher の自律実行トレース、右に Scraper Doctor の診断トレース］**

> そして重要なのは、**この同じ設計パターンを、プロダクトと運用の両方に使っている**ことです。
> 計画を立て、専門家を並列に走らせ、独立した批判役が検証し、人間に届く前にゲートを通す。決して自動で実行しない。
> Citify はこれを「どの街が自分に合うか」を考える Watcher に作り、
> **まったく同じ思想のまま、自分たちのスクレイパーがなぜ壊れたかを診断する Scraper Doctor / コスト異常を見張る Cost Hunter に再利用しています。**
> エージェントで作り、エージェントで運用する — これが私たちの「DevOps × AI Agent」の答えです。

### 想定 Q&A

**Q1: 政治的中立性は？**
> 3 層で担保。①プロンプトで賛否表明を禁止 ②出力後の禁止語・政治家名・党名検出で再生成 or ルールベース代替 ③Imagen は `person_generation="dont_allow"` + AI 生成ラベル + SynthID。全議題に原典リンクを併記します。

**Q2: スクレイピングの適法性は？**
> robots.txt と利用規約を必ず尊重します。実例として、大手議事録システムの 1 つは robots.txt が議事録パスを全面 Disallow だったため、**対応済みコードごと Drop しました**。全文転載はせず要約 + 原典リンクに統一しています。

**Q3: LLM コストは？**
> 翻訳・採点は Gemini 2.5 Flash 中心、フィードは in-memory キャッシュ + Cache-Control、Cloud Run は scale-to-zero、バッチは Cloud Scheduler で 1 日 1 回。さらにコスト異常を検知する Cost Anomaly Hunter エージェントを自前で運用しています。

**Q4: ADK を使った理由は？**
> 計画→並列専門家→自己検証の自律クルー（Watcher と Ops crew）、自律ツールループ（Concierge、translator/relevance を sub_agents に構成）、構造化アウトプットが標準化されているからです。一方、翻訳→採点→配信の定型パイプラインは ADK ではなく Pub/Sub + Cloud Run Jobs を選びました。**適材適所の使い分け**も設計判断として説明できます。

**Q5: 既存アプリとの違いは？**
> 一般の自治体ニュースアプリは「集約・配信」止まり。Citify は ①若者向け翻訳 ②年代×関心のパーソナル化 ③自治体比較 ④Watcher による自律分析、の 4 点で違います。特に③④は自治体 HP には構造的に提供できません。

**Q6: DevOps テーマへの貢献は？**
> Terraform で全リソース管理、GitHub Actions + Cloud Build の CI/CD、パスベース変更検知、Cloud Logging での観測。加えて**運用そのものをエージェント化**しています — スクレイパー失敗を診断・修正提案する Scraper Doctor、コスト異常検知の Cost Hunter。どちらも自動実行はせず人間レビュー前提の提案までです。そして両者は **Watcher と同一の設計パターン**（計画→並列専門家→独立批判→人間ゲート・自動実行なし）で作っています。

**Q7: 自分たちのセキュリティ姿勢はどう担保している？**
> 提出前に自己コードレビューのパスを回しました。実際に **CORS が `Origin` を反射しつつ資格情報を許可する設定（credentials 付きワイルドカード相当）の脆弱性**を自分で発見し、同日に web ドメイン限定 + credentials off に修正しています。あわせて高コストな LLM/エージェント endpoint（concierge・watcher/run）に**レート制限**を追加し、倫理フィルタ（政党名・氏名+役職の検出）を全生成エージェントに配線しました。CI では ruff / pytest / tsc / vitest / next build / terraform fmt に加え **gitleaks の全履歴シークレットスキャン**をゲートにしています。「見つけて直した脆弱性を、コミット単位で説明できる」ことを実運用配慮の証拠にします。
